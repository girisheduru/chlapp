- **Dashboard (Home):** See all habits, streaks, stones, and quick access to check-in or reflect.

- **Check-in:** Log “I did it” (or not today), add a stone to the jar, keep streaks, and optionally note obstacles/helpers.
- **Reflection:** Weekly insights, reflection questions, and small experiment suggestions. Content is generated by a **ReAct agent** that can use **Tavily** web search (e.g. James Clear / Atomic Habits) plus the LLM to produce personalized, Atomic Habits–aligned content.

- **App (React / Vercel):** Frontend — what the user sees and taps (dashboard, onboarding, check-in, reflection). Built with React; deployed on Vercel.
- **API (FastAPI, Python / Railway):** Backend — receives and sends habits, check-ins, streaks, and reflection requests. Built with FastAPI (Python); deployed as a Railway project.
- **MongoDB:** Database — stores habits, preferences, check-in history, streaks, and cached reflection items.
- **OpenAI models:** Used to generate identity/habit/cue options in onboarding. Also used by the ReAct agent when generating reflection content.
- **ReAct agent:** Generates reflection content (insights, questions, experiment suggestions). Uses reasoning and tool use: it can call Tavily to search for James Clear / Atomic Habits content, then synthesize that with the user’s habit context into personalized reflection items. Results are cached so the Reflection screen loads quickly.
- **Tavily:** Web search tool used by the ReAct agent to pull in up-to-date or authoritative content (e.g. Atomic Habits) when building reflection insights and experiments.
- **Comet Opik:** AI evals and observability — we use Opik to trace and evaluate the AI parts of the application (LLM calls, reflection agent).

### 3.1 How reflection content is generated

Reflection insights, questions, and experiment suggestions are produced by a **ReAct agent** that can use **Tavily** to search the web (e.g. for James Clear / Atomic Habits content), then uses the LLM to turn that plus the user’s habit context into structured content.

- **ReAct agent:** Reasons over the user’s habit and streak data, optionally calls **Tavily** to fetch relevant material, then calls the **LLM** to produce insights, reflection questions, and three experiment suggestions (anchor, environment, enjoyment). Output is cached in MongoDB for fast loads.

## 4. Main product flows (what happens when)

| Flow | Trigger | What happens |
|------|--------|---------------|
| **Onboarding** | User adds a new habit | User picks or edits identity, nucleus habit, supernova habit, cue, enjoyment, environment. AI can suggest options at each step. Plan is saved. |
| **Check-in** | User taps “Check in” on a habit | User chooses what they did (nucleus, supernova, other, or “not today”). One stone added when they did something. Streak and totals update. Optionally they note obstacles/helpers. |
| **Reflection** | User opens Reflection for a habit | App loads cached or fresh content. When fresh, a **ReAct agent** runs: it may use **Tavily** to search for James Clear / Atomic Habits content, then the LLM produces insights, reflection questions, and 3 experiment suggestions (anchor, environment, enjoyment). Results are cached. User can answer questions and apply suggestions to their plan. |
| **Background prep** | After a check-in | System may pre-generate reflection content for that habit (via the ReAct agent and Tavily when configured) so the Reflection screen loads quickly next time. |

## 5. Key product concepts

- **Identity:** “I am someone who…” — the identity statement the user is building with this habit.
- **Nucleus habit:** Smallest “show up” version (e.g. 2 minutes).
- **Supernova habit:** Full version when they have more energy.
- **Cue / anchor:** When or after what the habit happens (e.g. “after morning coffee”).
- **Stones & jar:** One stone per successful check-in; visual progress.
- **Streak:** Consecutive days with at least one check-in.
- **Reflection:** Weekly insights and experiment suggestions, grounded in Atomic Habits–style ideas.
